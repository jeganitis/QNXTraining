Requirements Specification

A hybrid QNX IPC system:
message + pulse + shared memory

Defines a producer/consumer frame-transfer service in which control negotiation uses synchronous native messages, event notification uses pulses, and bulk data moves through zero-copy POSIX shared memory — each mechanism doing only the job it's suited for.

Doc ID: QNX-HYBRID-IPC-01
Status: Draft for review
Target: QNX Neutrino, SMP
Owners: Producer (data source), Consumer (supervisor)
1.

Context & problem statement

A producer process generates fixed-size data frames (e.g. sensor/camera frames, log batches) at a steady rate and must hand them to a consumer process for further work. Two naive designs fail: routing every frame through a synchronous MsgSend() forces a full kernel copy per frame and stalls the producer until the consumer replies; routing everything through pulses alone is unsafe, since a pulse cannot carry a variable-size payload. The system therefore needs all three mechanisms, each scoped to what it does well.

2.

Mechanism responsibilities

Each IPC primitive is assigned exactly one job in this system — no overlap, no redundancy.

Native message

Connection setup & control

One-time, blocking MsgSend()/MsgReceive()/MsgReply() rendezvous used to negotiate the shared memory region: object name, size, and buffer count. Also used for orderly shutdown requests.

Used at: session start, session end
Pulse

Asynchronous event signaling

Fixed-size, non-blocking notifications: FRAME_READY from producer to consumer when a buffer is filled, BUFFER_FREE from consumer back to producer when it's safe to reuse a slot.

Used at: every frame, steady-state loop
Shared memory

Bulk zero-copy data plane

shm_open() + mmap() region holding a double-buffered ring of frame slots. The actual frame bytes never pass through the kernel's message-copy path.

Used at: every frame, steady-state loop
3.

Functional requirements

Each requirement is tagged with the IPC mechanism it governs.

IDRequirementMechanism
FR-01 The Consumer shall create a named channel via name_attach() before accepting any connection, and block on MsgReceive() until a request arrives. message
FR-02 The Producer shall connect via name_open() and issue exactly one synchronous MsgSend() control request before any data transfer, specifying requested frame size and buffer count. message
FR-03 The Consumer shall allocate the shared memory object (shm_open() + ftruncate()) sized to hold N double-buffered frame slots, and return the object name, size, and slot layout to the Producer in its MsgReply(). message shm
FR-04 The Producer shall mmap() the shared memory object using the name and size received in FR-03 before writing any frame data. shm
FR-05 The Producer shall write each frame directly into the next free slot in the shared region and shall not copy frame data through any message primitive. shm
FR-06 After a frame write completes, the Producer shall notify the Consumer with MsgSendPulse(FRAME_READY, slot_index) and shall not block waiting for acknowledgement. pulse
FR-07 The Consumer shall distinguish pulses from blocking requests on the same channel using rcvid == 0, and dispatch on msg.pulse.code. pulse
FR-08 On receiving FRAME_READY, the Consumer shall read the indicated slot directly from the shared region and shall not request the data via a separate message. shm
FR-09 After consuming a slot, the Consumer shall notify the Producer with MsgSendPulse(BUFFER_FREE, slot_index) so the slot may be reused, enabling double-buffered pipelining. pulse
FR-10 The Producer shall not write into a slot until it has received a corresponding BUFFER_FREE pulse for that slot (or it is the slot's first use). pulse shm
FR-11 Session shutdown shall use a final pulse (SESSION_DONE) followed by the Consumer unmapping and shm_unlink()-ing the region, then name_detach(). pulse shm
FR-12 The Consumer shall handle _PULSE_CODE_DISCONNECT by calling ConnectDetach() to release kernel resources if the Producer terminates unexpectedly mid-session. pulse
4.

Architecture overview

Control and event paths run through the kernel's message-passing subsystem; the data path runs directly between the two processes' mapped address spaces.

7.

Non-functional requirements

IDRequirement
NFR-01Steady-state frame handoff shall not invoke any kernel message copy larger than the fixed pulse payload (5 bytes) — bulk data movement cost shall be limited to the producer's write and consumer's read of shared memory.
NFR-02The double-buffer handshake (FR-10) shall be the sole mutual-exclusion mechanism for slot access; no additional lock/semaphore is required as long as strict alternation is enforced.
NFR-03Control setup (FR-01–FR-03) shall complete once per session and shall not reoccur in the per-frame steady-state path.
NFR-04The system shall remain correct if the Producer disconnects mid-session (FR-12); the Consumer shall not deadlock waiting on a pulse that will never arrive.
NFR-05All pulse codes shall be defined above the reserved system range to avoid collision with _PULSE_CODE_DISCONNECT / _PULSE_CODE_UNBLOCK.
