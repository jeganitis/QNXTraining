Pulses: the non-blocking half
of QNX message passing

A live walkthrough of Worker.c and Supervisor.c — one process sending fixed-size, fire-and-forget notifications to another over a named channel, and reading its PID back out with ConnectClientInfo().

What a pulse is

A tiny, fixed-format kernel message — one byte code + a 4-byte value. No reply is expected or possible. The sender never blocks waiting for the receiver.

Why not MsgSend()

MsgSend() blocks the caller until MsgReply() — a synchronous rendezvous. A heartbeat shouldn't stall the worker's loop, so it uses MsgSendPulse() instead: queue it and move on.

How it's told apart

MsgReceive() returns rcvid == 0 for a pulse, versus a positive rcvid for a real blocking message that expects MsgReply().

The code being simulated

// Connect to the supervisor's named channel.
int coid = name_open("sup_pulse", 0);
if (coid == -1) { perror("name_open(sup_pulse)"); return 1; }

// Send a few heartbeats.
for (int i = 1; i <= 5; ++i) {
    MsgSendPulse(coid, 10, PULSE_HEARTBEAT, i);
    printf("[worker] sent heartbeat %d\n", i);
    usleep(200 * 1000); // 200 ms
}

// Tell supervisor we're done.
MsgSendPulse(coid, 10, PULSE_DONE, 0);
name_close(coid);
// Create a named channel so workers can connect.
name_attach_t *att = name_attach(NULL, "sup_pulse", 0);

for (;;) {
    int rcvid = MsgReceive(att->chid, &msg, sizeof(msg), NULL);

    if (rcvid == 0) {  // it's a pulse
        switch (msg.pulse.code) {
        case _PULSE_CODE_DISCONNECT:
            ConnectDetach(msg.pulse.scoid);
            break;

        case PULSE_HEARTBEAT: {
            ConnectClientInfo(msg.pulse.scoid, &ci, 0);
            printf("[supervisor] heartbeat from pid=%d val=%d\n", ci.pid, msg.pulse.value.sival_int);
            break;
        }

        case PULSE_DONE: {
            ConnectClientInfo(msg.pulse.scoid, &ci, 0);
            printf("[supervisor] worker pid=%d DONE. Exiting.\n", ci.pid);
            name_detach(att, 0);
            return 0;
        }
        }
    }
}
